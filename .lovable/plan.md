# Gotham OS Branded Email System — Build Plan

This is a large, multi-phase build. I'll deliver it in 5 phases so we can verify behavior between each, instead of one giant unverifiable drop.

---

## Phase 1 — Brand foundation + alert↔email sync layer

**Shared brand kit** (`src/lib/email-templates/_brand.tsx` — already exists, extending):
- Lock palette: ink `#0B0B0B`, graphite `#1A1A1A`, gold `#C8922A`, off-white `#F7F3EA`, muted `#7A7A7A`.
- Add reusable building blocks: `<BrandLayout>`, `<StatusBadge variant="success|warning|critical|info">`, `<DataRow label value>`, `<DetailCard>`, `<PrimaryButton>`, `<MetaFooter>`.
- Standard structure: dark header (Gotham OS / Operational Notification) → gold rule → white content card with status badge + key/value table → CTA button → off-white footer ("Generated by Gotham OS").

**Alert ↔ email sync schema:**
- Extend `alerts` with `email_status` enum (`none | queued | sent | failed | suppressed`), `email_message_id`, `email_sent_at`, `email_error`.
- Add `email_send_log.alert_id` FK so every send links back to its alert.
- New table `notification_preferences` (user_id, channel_email_enabled, frequency `immediate | daily_digest | critical_only`, per-category opt-outs jsonb).
- Helper server fn `enqueueAlertEmail(alertId, templateName, templateData, recipients[])` that:
  1. Reads each recipient's preference + suppression list.
  2. Calls `enqueue_email` on `transactional_emails` pgmq queue.
  3. Updates alert with `email_status='queued'`.
  4. On send-route success/failure, updates alert + writes `email_send_log` row keyed by `(alert_id, recipient_email)`.

**Recipient resolver helpers** (`src/lib/email/recipients.server.ts`): `getOwners()`, `getManagersForTrailer(trailerId)`, `getEmployee(userId)`, `getCrewForTrailer(trailerId)` — all honoring `profiles.active` and `notification_preferences`.

## Phase 2 — All 22 branded templates

One file per template under `src/lib/email-templates/`, all using the shared brand kit. Each registered in `registry.ts` with typed `templateData` props and `previewData` for the dashboard preview.

Grouped by module:

| Module | Templates |
|---|---|
| Schedule | `schedule-published`, `schedule-submitted`, `schedule-approved`, `schedule-locked`, `schedule-changed` |
| Time clock | `time-adjustment-request`, `time-adjustment-approved`, `time-adjustment-declined`, `missed-clock-out`, `auto-clock-out` |
| Inventory | `inventory-order-submitted`, `inventory-order-approved`, `inventory-order-declined`, `low-stock-alert` |
| Cash | `cash-drawer-submitted`, `cash-variance-alert`, `cash-drop-submitted` |
| Training | `training-assigned`, `training-completed` |
| Operations | `daily-recap-submitted`, `announcement-published`, `critical-alert` |

Each template renders the exact data points listed in the request (employee name, week range, shift list, totals, variance amounts, items, etc.) plus a single primary CTA deep-linking into Gotham OS (`/schedule/$id`, `/inventory/orders/$id`, `/cash/drawers/$id`, etc.).

## Phase 3 — Module integrations (triggers)

Hook every module's existing alert-creation path through `enqueueAlertEmail`. No new business logic — reuse the existing triggers in the database where possible, surface them via server functions that the app already calls.

- **Schedule**: in `src/lib/schedule.functions.ts` — on submit → owner alert + email; on approve → publish alert + per-employee emails; on lock/change → manager + crew emails.
- **Time clock**: extend `run_daily_rollover` SQL function to insert alerts for any `auto_closed` punch; new server fn for adjustment requests (create alert + email owner; on decision, email employee).
- **Inventory**: existing `emit_inventory_order_alert` trigger already creates the alert — add a Postgres `AFTER INSERT` trigger on `alerts` (type `inventory_order`) that calls a server route `/api/public/hooks/alert-email-dispatch` via `pg_net` to enqueue the branded email. Same dispatcher handles owner-decision alerts back to manager.
- **Cash**: on `cash_drawer_sessions` status → `submitted` insert alert; if `variance` non-zero mark `critical`; dispatcher sends owner email + PDF link.
- **Training / Recap / Announcement / Critical alert**: existing alert insertions already wired; dispatcher picks them up by `alerts.type`.

**Central dispatcher** (`src/routes/api/public/hooks/alert-email-dispatch.ts`): receives `{alert_id}` from pg_net, looks up alert + module-specific data, picks template, resolves recipients, calls `enqueueAlertEmail`. Idempotent on `alert_id+template` (uses `email_send_log` uniqueness).

## Phase 4 — Preferences UI + delivery tracking

- **Settings page** `src/routes/_authenticated/settings/notifications.tsx`: toggle email on/off, choose frequency, per-category checkboxes (schedule, time clock, inventory, cash, ops). Owners can edit anyone's; users edit their own.
- **Delivery log view** under Owner Dashboard → "Email Activity": table from `email_send_log` joined with `alerts` showing recipient, subject, template, source module, status badge, sent time, retry count, failure reason. Filters by date/module/status. Resend action for `failed` rows.
- Daily digest worker: cron at 7am local trailer time aggregates the previous day's `digest`-frequency alerts per user into a single `daily-digest` template.

## Phase 5 — QA + verification

Automated server-side test script (`scripts/qa-email-flow.ts`, invoked via server fn) that walks all 12 listed scenarios end-to-end against a seeded test trailer, then asserts:
- Alert row created with correct type/priority/assigned_role.
- Email enqueued, then `sent` in `email_send_log` within 30s.
- `alerts.email_status = 'sent'`, message_id matches.
- No duplicate sends for the same `(alert_id, recipient)`.
- Dashboards reflect new state (queried via server fn).
- History (`audit_log` / `change_log`) has matching row.

I'll run this script after Phase 4 and surface the report (pass/fail per scenario + screenshots of two key emails rendered in the preview route) before marking done.

## Technical notes (for reference)

- All sends go through the existing pgmq `transactional_emails` queue + `/lovable/email/queue/process` cron — no new queue infra.
- Recipient resolution and template rendering happen at enqueue time so a paused queue or DNS hiccup never loses data.
- All triggers funnel through one dispatcher = one place to add idempotency, suppression check, and preference check.
- Email preferences live in a new `notification_preferences` table with RLS (self-read/write; managers read trailer crew; owners read all).
- Brand kit uses inline styles only (email-safe) and embeds DM Sans via `@react-email/components` `<Font>`.
